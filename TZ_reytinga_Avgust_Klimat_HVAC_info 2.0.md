 # Техническое задание
## Подсистема рейтинга кондиционеров и расчёта индекса «Август-климат» для HVAC-info

## 1. Цель проекта

Разработать подсистему HVAC-info, которая позволяет:

- хранить методику индекса «Август-климат»;
- вводить и верифицировать данные по моделям кондиционеров;
- рассчитывать индекс модели по утверждённой методике;
- публиковать рейтинг моделей в разных режимах отображения;
- масштабировать систему на разные рынки и языки;
- предоставлять структурированные данные внешним ИИ-агентам для подбора кондиционеров.

---

## 2. Область применения

Первая версия рейтинга распространяется на:

- бытовые настенные сплит-системы;
- модели с холодопроизводительностью до 4,0 кВт.

Архитектура должна допускать расширение на:

- другие диапазоны мощности;
- другие типы климатического оборудования;
- другие методики и региональные версии индекса.

---

## 3. Основная бизнес-логика

Индекс «Август-климат» — это интегральная числовая оценка модели кондиционера, рассчитанная на основе набора критериев, сгруппированных по тематическим блокам.

Система должна поддерживать:

- редактируемые критерии;
- редактируемые веса;
- разные типы шкал;
- региональные версии критериев;
- три режима рейтинга;
- разные источники данных;
- лабораторную и редакционную верификацию.

---

## 4. Режимы рейтинга

На первом этапе не разрабатываются отдельные режимы рейтига

## 5. Объект оценки

Объектом расчёта является конкретная модель кондиционера 

### 5.1. Обязательные поля карточки модели
- бренд;
- серия;
- модель внутреннего блока
- модель наружного блока;
- номинальная холодопроизводительность;
- тип оборудования;
- регион доступности;
- статус публикации;
- ссылки на источники данных.

### 5.2. Региональная доступность модели
Для модели должны храниться признаки:
- доступна в RU;
- доступна в EU.

## 6. Формула расчёта индекса

Базовая формула:

**Index = Σ(Weight_i × Score_i) / 100**

Где:
- `Weight_i` — вес параметра в процентах;
- `Score_i` — нормированный балл параметра в диапазоне 0–100.

### 6.1. Требования к весам
- сумма весов всех параметров в активной методике должна быть ровно **100%**;
- веса групп на первом этапе равны сумме весов входящих в них параметров;
- система должна контролировать это правило валидацией.

### 6.2. Пересчёт рейтинга
После изменения весов или формул:
- система должна предупредить пользователя о необходимости пересчёта;
- пересчёт запускается вручную оператором через кнопку/действие в админке.

## 8. Шкалы оценки

### 8.1. Универсальная шкала для категориальных параметров

### 8.2. Индивидуальные шкалы

---

## 9. Формула числовых параметров

Для числовых параметров, где используются `min / median / max`, принять модель расчёта:

- `min` соответствует нижней опорной точке;
- `median` — опорная точка внутри формулы;
- `max` — верхний эталон;
- значения выше `max` получают `Score = 100`;
- в интерфейсе такие значения помечаются как **«выше эталона»**.

### 9.1. Общий принцип
Для числовых параметров с min/median/max используется вариант логики **C**:
- роль медианы встроена в формулу;
- значения выше эталона визуально отмечаются;
- Score выше 100 не допускается.

## 12. Специальные правила по отдельным параметрам

### 12.1. Площадь теплообменника внутреннего блока
Использовать логику:
- по классу мощности;
- с учётом медианы;
- с меткой «выше эталона» при превышении max.

### 12.2. Мощность компрессора
Это отдельный специальный параметр с формульной логикой.

#### Правило сравнения
Сравнивается:
- каталожная холодопроизводительность кондиционера;
- refrigeration capacity компрессора.

#### Интервальная шкала
- `< 80%` → 0–30
- `80–90%` → 50
- `90–95%` → 70
- `95–100%` → 90
- `101–110%` → 100
- `>110%` → 100

#### Отсутствие данных
Это единственный параметр, где отсутствие значения **не блокирует публикацию**, так как используется fallback:

- японский бренд = 90
- китайский/OEM = 50

Значения fallback должны быть редактируемыми в CMS.

#### Справочник происхождения бренда
Для работы fallback-схемы нужен справочник с классификацией брендов по происхождению/типу для данного правила. Категорию присваивает **только администратор**.

### 12.3. Работа при сниженном питающем напряжении
Отдельный параметр со шкалой:

- 100В = 100
- 150В = 70
- 230В = 0

Параметр является **строго лабораторным**.
При отсутствии значения модель **не публикуется** в лабораторном рейтинге.
Принадлежность параметра к лабораторным должна настраиваться в CMS.

### 12.4. Ионизатор
- полноценный ионизатор = 100
- упрощённый / маркетинговый = 50
- нет = 0

### 12.5. УФ-лампа
- крупная лампа = 100
- маленькая лампа / светодиод = 50
- нет = 0

### 12.6. Фильтры тонкой очистки
- два и более полноценных фильтра = 100
- один фильтр = 50
- нет = 0

## 13. Лабораторные параметры

Система должна уметь помечать параметр как лабораторный через CMS.
Нельзя зашивать этот признак только в код.

### 13.1. Статусы лабораторного значения
Для лабораторных параметров должны поддерживаться статусы:
- не измерено;
- ожидает измерения;
- не участвует в данном режиме.

Это должно отличаться от нулевого балла.

## 14. Источники данных

Для каждого значения параметра должны храниться:

- числовое или категориальное значение;
- источник;
- ссылка на источник;
- текстовый комментарий к источнику;
- статус верификации;
- дата ввода;
- пользователь, внёсший данные;
- пользователь, утвердивший данные.

### 14.1. Публикация источника
Архитектура должна предусматривать возможность публикации ссылки на источник в пользовательском интерфейсе, в том числе рядом с визуальным блоком/изображением.

---

## 15. Верификация данных

Для каждой модели и/или значения параметра должна поддерживаться верификация.
Минимальные статусы:

- данные по каталогу;
- проверено редакцией;
- подтверждено лабораторией.

Этот блок должен быть:
- доступен для внутренней работы;
- опционально выводим на публичной карточке модели;
- управляем через настройку публикации.

---

## 16. Версионирование методики и расчётов

Система должна поддерживать внутренние версии методики.

### 16.1. Требования
- версия методики хранится в системе;
- публично её можно не показывать;
- старые результаты расчётов должны храниться;
- после обновления методики пересчёт выполняется на актуальной версии;
- история предыдущих расчётов не удаляется.

---

## 17. История изменений

Для моделей, параметров и расчётов должна вестись история изменений.

### 17.1. Что логировать
- кто изменил;
- когда изменил;
- старое значение;
- новое значение;
- комментарий к изменению;
- на какой сущности сделано изменение.

- указание статуса публикации.

### 18.3. Раздел «Расчёты»
- пересчитать одну модель;
- пересчитать все модели по выбранной методике;
- увидеть предупреждение после изменения формул/весов;
- запускать пересчёт вручную.

### 18.4. Раздел «Журнал»
- смотреть историю изменений;
- фильтровать по модели, параметру, пользователю, периоду.
